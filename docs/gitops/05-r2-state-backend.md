# Step 5 - Cloudflare R2 state backend

OpenTofu state is stored in [Cloudflare R2](https://www.cloudflare.com/products/r2/) (S3-compatible object storage).

## Why R2

- **Free tier**: 10 GB storage, no egress fees.
- **S3-compatible**: works with OpenTofu's built-in S3 backend.
- **No separate state locking** needed for a single-user homelab. If you need locking later, R2 can pair with a DynamoDB-compatible alternative.

## Prerequisites

- A Cloudflare account.
- R2 enabled in the Cloudflare dashboard.

## Steps

### 1. Create an R2 bucket

In the Cloudflare dashboard, navigate to **R2** and create a bucket named:

```
autolab-opentofu-state
```

The bucket name must be globally unique across all R2 buckets.

### 2. Create R2 API tokens

In the R2 dashboard, go to **Manage R2 API Tokens** and create a new token with **Object Read & Write** permissions. Save the following immediately — they are shown only once:

- **Access Key ID** → will be stored as `R2_ACCESS_KEY_ID`
- **Secret Access Key** → will be stored as `R2_SECRET_ACCESS_KEY`

### 3. Get your Cloudflare account ID

In the R2 dashboard URL you will see something like:

```
https://dash.cloudflare.com/YOUR_ACCOUNT_ID/r2/...
```

Copy the account ID — you need it for the S3 endpoint.

### 4. Terramate backend generation

The backend configuration is generated by Terramate from `infra/_base/backend.tm.hcl`. Non-secret defaults are generated into `versions.tf`, while the bucket name and endpoint still come from the R2 setup inputs. The `key` uses `terramate.stack.path.relative` so each stack gets its own state file automatically. The generated `versions.tf` will contain:

```hcl
backend "s3" {
  bucket  = "autolab-opentofu-state"
  key     = "${terramate.stack.path.relative}/terraform.tfstate"
  region  = "auto"

  skip_credentials_validation = true
  skip_requesting_account_id  = true
  skip_metadata_api_check     = true
  skip_region_validation      = true
}
```

### 5. Set local environment variables

Before running `tofu init` locally, set these env vars:

```bash
export AWS_ACCESS_KEY_ID=<your-r2-access-key-id>
export AWS_SECRET_ACCESS_KEY=<your-r2-secret-access-key>
```

OpenTofu uses the standard `AWS_` environment variables to authenticate against the S3-compatible endpoint.

### 6. Initialize OpenTofu

Run Terramate generate first, then init with the endpoint via CLI:

```bash
cd infra
terramate generate
cd stacks/lab
tofu init -backend-config="endpoint=https://YOUR_ACCOUNT_ID.r2.cloudflarestorage.com"
```

Tofu will detect the S3 backend and prompt you to copy existing state if any. Confirm to complete the migration.

## Lifecycle and maintenance

- R2 bucket creation is idempotent in the helper scripts: rerunning setup should reuse the existing bucket rather than create a new one.
- Setup scripts never delete the bucket. Removing the bucket is a manual operator action only.
- Token rotation: create a new R2 API token with the same permissions, update `R2_ACCESS_KEY_ID` and `R2_SECRET_ACCESS_KEY` in GitHub Environments, then update any local `AWS_*` env vars and re-run `tofu init` if needed.

## Store credentials in GitHub

Add the R2 credentials as GitHub Environment secrets (see [Step 6 - GitHub Environments](./06-github-environments.md)):

| Secret | Value |
|--------|-------|
| `R2_ACCOUNT_ID` | Your Cloudflare account ID (used to construct the R2 endpoint URL) |
| `R2_ACCESS_KEY_ID` | The R2 access key ID from step 2 |
| `R2_SECRET_ACCESS_KEY` | The R2 secret access key from step 2 |

The GitHub workflows map these to `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` when running `tofu plan` or `tofu apply`.

## State locking

State locking is **not configured**. For a single-user homelab, concurrent applies are unlikely. If you add a team or automate frequent concurrent runs later, you can add state locking with a DynamoDB-compatible locking mechanism. R2 itself does not provide locking.

## Automated setup

A script can create the R2 bucket and API tokens for you:

```bash
bash scripts/setup-r2-backend.sh --account-id YOUR_ACCOUNT_ID
```

See [scripts/setup-r2-backend.sh](../../scripts/setup-r2-backend.sh) for options including `--dry-run` and `--bucket-name`.

The setup script calls `r2-create-token.sh --format json` and parses credentials with `jq`. The token script still defaults to the human-readable `ACCESS_KEY_ID=...` text output for direct use, but automation uses the structured output to avoid fragile line parsing.

R2 defaults are generated from `infra/r2-config.yaml` into both Terramate globals and shell defaults. Regenerate all artifacts with:

```bash
bash scripts/generate-r2-config.sh
```

For targeted drift checks, use:

```bash
bash scripts/generate-r2-config.sh --terramate --check
bash scripts/generate-r2-config.sh --defaults --check
```
